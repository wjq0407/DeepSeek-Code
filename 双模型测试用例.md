# 双模型协作架构 — 测试用例

> 每个用例说明：**测试哪个阶段、输入什么 prompt、预期观察什么。**
> 所有代码写入当前工作目录（不碰 src/），建议建一个 `test_gen/` 子目录集中存放测试产出。
> 按顺序测试，每个用例后观察 Agent 的工具调用链。

---

## 测试 1：写复杂代码 → 自动触发深度审查（阶段1）

```
帮我写一个 TypeScript 的 RetryQueue 类（重试队列），写在 test_gen/retry-queue.ts，要求：
- 支持添加任务（异步函数），失败后自动重试
- 支持配置最大重试次数、重试间隔（支持固定和指数退避两种策略）
- 支持并发控制（最多同时执行 N 个任务）
- 支持暂停/恢复/清空队列
- 有完整的状态枚举（idle/running/paused/drained）
- 事件驱动，每个任务完成/失败/重试时触发回调
```

**预期观察：**
1. Flash 调用 write_file（>30 行）
2. 下一轮**自动调用 review_code**（无需用户说"帮我审查"）← 阶段1 效果
3. Pro 输出中文审查报告（风险等级、问题清单、建议）
4. Flash 根据审查结果修正代码

**验证点：** ✅ write_file 后自动触发 review_code

---

## 测试 2：写短代码 → 用快速验证（阶段2 verify_code）

```
帮我在 test_gen/retry-queue.ts 里加一个 stats() 方法，返回 { total, success, failed, retried, pending } 统计对象
```

**预期观察：**
1. Flash 调用 edit_file（<30 行改动）
2. 调用 **verify_code**（不是 review_code）← 阶段2 效果
3. PASS → 继续 / FAIL → 修正后复查

**验证点：** ✅ 小改动走轻量验证，不走深度审查

---

## 测试 3：多轮操作后自动审核最终答复（阶段3 Elevate 闸门）

先执行测试 1 和 2，确保 test_gen/retry-queue.ts 存在。然后问：

```
帮我总结一下 test_gen/retry-queue.ts 的设计：它解决了什么问题、有哪些关键设计决策、有没有潜在风险？
```

**预期观察：**
1. Flash 读文件、分析代码
2. 准备输出分析结论时 **Elevate 闸门触发**（loop 硬拦截）← 阶段3 效果
3. 系统自动注入 "先调 verify_answer"，Flash 调用 verify_answer 审核
4. Pro 审核通过后 Flash 才输出最终答复

**验证点：** ✅ 多轮工具操作后，最终答复必经 verify_answer

## 测试 4：简单查询不触发任何 Pro 工具（成本优化）

```
帮我读一下 package.json，告诉我这个项目名字是什么、用了什么构建工具
```

然后对比：

```
帮我解释一下 JavaScript 里面 Promise、async/await 和事件循环的关系
```

**预期观察：**
- 两个都应该 Flash 直通，零 Pro 调用
- 不触发 review_code / verify_code / verify_answer / deep_gen / Elevate

**验证点：** ✅ 简单查询最低成本，不浪费 Pro

## 测试 5：复杂架构任务 → deep_gen 深度生成（阶段4）

```
帮我设计一套「文件上传进度追踪系统」的完整方案。需求：
- 前端上传大文件（支持分片），后端接收并组装
- 需要 WebSocket 实时推送上传进度
- 支持断点续传（客户端记录已上传分片）
- 支持并发上传控制（最多 3 个文件同时上传）
- 考虑大文件（>1GB）的内存和磁盘管理

先用 deep_gen 生成完整技术方案（task_type=plan），我再决定要不要实现。
```

**预期观察：**
1. Flash 判断这是复杂架构任务
2. Flash 调用 **deep_gen**（task_type=plan）← 阶段4 效果
3. Pro 深度生成分步方案（目标/步骤/风险/技术选型）
4. Flash 解读方案呈现给用户

**验证点：** ✅ 复杂任务 Flash 委托 Pro 深度生成

## 测试 6：安全敏感代码 → 全链路质量保障（阶段1+2+3 联动）

```
帮我写一个 Node.js 的 TokenBucket 限流器，写在 test_gen/token-bucket.ts，要求：
- 基于令牌桶算法，支持突发流量
- 使用 HMAC-SHA256 对令牌做签名防伪造
- 令牌存储用 Map，但要考虑内存泄漏（定期清理过期桶）
- Key（签名密钥）从环境变量 TOKEN_BUCKET_SECRET 读取，必须有 fallback 警告
- 不允许在任何日志中输出原始令牌内容
```

**预期观察：**
1. Flash write_file（>30 行 + 安全敏感）
2. → **自动触发 review_code**（阶段1：安全代码强制审查）
3. → 可能触发 verify_code（阶段2：快速检查）
4. → 审查发现问题 → 修正 → 再审
5. → 最终输出前 **Elevate 闸门**（阶段3：多轮操作审核）
6. → verify_answer 审核后输出

**验证点：** ✅ 安全敏感代码走过完整「写→查→修→再查→审→输出」质量链

## 测试 7：成本意识测试

```
帮我写一个简单的工具函数：把字符串转成 URL 友好的 slug（小写、去特殊字符、空格换连字符），写在 test_gen/slugify.ts
```

然后紧接着问：

```
现在帮我全面分析 test_gen/ 目录下所有文件，从代码质量、安全性、性能三个维度给出评估报告
```

**预期观察：**
- 测试 A：Flash 直通（<20 行简单函数），零 Pro 调用
- 测试 B：Flash 调用 review_code 深度审查（多文件 + 用户要求）

**验证点：** ✅ Flash 能正确区分「简单任务自己做」和「复杂任务调 Pro」

## 观察清单

测试时观察工具调用链，判断改造是否生效：

| 场景 | 预期调用链 | 对应阶段 |
|------|-----------|:--------:|
| 写超过 30 行代码 | write_file → review_code | 阶段1 |
| 小改动 | edit_file → verify_code | 阶段2 |
| 多轮操作后输出 | ... → Elevate → verify_answer → 输出 | 阶段2+3 |
| 复杂架构设计 | Flash 判断 → deep_gen → Pro 生成 | 阶段4 |
| 简单查询 | Flash 直通，零 Pro | 阶段4(成本) |
| 安全敏感代码 | write → review → fix → verify → elevate → verify_answer | 全链路 |

### 调用链速记

```
review_code   = Flash 写了代码后 Pro 在审（被动触发=阶段1 工作）
verify_code   = Flash 快速自查（轻量=阶段2 工作）
verify_answer = Pro 审核 Flash 准备给用户的答复（质量门=阶段2+3 工作）
deep_gen      = Pro 直接生成复杂内容（深度生成=阶段4 工作）
Elevate 闸门   = loop.ts 硬拦截，强制先调 verify_answer 再输出（阶段3 工作）
```
