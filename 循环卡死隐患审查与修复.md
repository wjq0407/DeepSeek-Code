# deepseek-code-agent · 循环相关「崩/卡死」隐患审查与修复

> 审查日期：2026-07-13
> 范围：与 Agent 主循环（`runAgent` 的 `while` 循环）相关的、会导致**整个程序崩溃或永久卡死**的隐藏 Bug。
> 方法：通读 `loop.ts / deepseek.ts / history.ts / tools/index.ts / mcp/* / agent/session.ts / agent/subagent.ts / cli/main.ts / cli/app.tsx`，并对经典死锁模式（无超时 await、信号未透传、进程树未回收）做重点核查。
> 验证：`tsc --noEmit` 通过；`run_command` 进程树清理已用真实进程实测通过。

---

## 一、已修复的高危隐患（会卡死 / 崩）

### 🔴 1. `history.compact()` 摘要压缩无 signal + 无超时（最高危，与上轮同类）
- **位置**：`src/agent/loop.ts:413` `await opts.history.compact()` → `src/context/history.ts:234` `this.client!.complete(...)`
- **机制**：主循环每轮工具执行后会 `await compact()` 做上下文压缩。当启用 LLM 摘要压缩时，内部 `complete()` **既没传 signal、也没超时**。
- **死锁场景**：用户在「工具跑完、即将进入 compact」的窗口期按 Ctrl+C → `signal.aborted=true`，但 runAgent 正阻塞在 `await compact()`；compact 内部的 `complete()` 不监听 abort，**UI 永久卡在"压缩中"，再按 Ctrl+C 也救不回**（信号没传到 compact）。
- **修复**：`compact(options?: { signal })` 透传 signal；`summarizeCompact` 调 `complete` 时带 `signal` + `timeoutMs: 60_000`。

### 🔴 2. 主循环 `streamChat` 流式 API 无 timeout
- **位置**：`src/llm/deepseek.ts:117` 流式 `create(...)` + `loop.ts:200`
- **机制**：`for await (const chunk of stream)` 依赖服务端持续推送。若连接建立后服务端不发包（网络中间层保活、服务端假死），该循环**永久挂起**。原代码只传了 `signal`（可手动中断），没有硬超时兜底。
- **修复**：`streamChat` 加 `timeout: 300_000`（SDK 原生）。超时走 `error` 事件、Ctrl+C 走 `aborted` 事件，区分清晰。

### 🔴 3. `complete()`（复合工具 + 记忆萃取）无 signal + 无超时
- **位置**：`deepseek.ts:205` `complete()`；调用点 `review.ts / audit.ts / discovery.ts / git.ts / memory/extractor.ts`
- **机制**：`review_code / audit_dependencies / terminology / project_discover / git_commit_msg` 以及会话结束的偏好萃取都走 `complete()`。原调用**全部不带 signal、不带超时**。
- **死锁场景**：执行复合工具期间用户按 Ctrl+C → 主循环 `await def.execute` 阻塞在 `complete()`，signal 没透传 → UI 卡在"执行 xxx 中"，Ctrl+C 救不了。
- **修复**：`complete` 加 `timeoutMs`（默认 180s，萃取 60s）；所有调用点透传 `ctx.signal`。

### 🔴 4. delegate 子 Agent 不可中断 + `whenDone` 永久挂起（连锁死锁）
- **位置**：`src/agent/subagent.ts` `execute` 丢弃 `ctx`；`cli/main.ts` 注入的 runner 未传 signal；`src/agent/session.ts` `runBackground` 的 `runOpts` 无 signal、`whenDone` 不监听 abort
- **机制**：`delegate` 工具执行时 `await opts.runner(task)` → `sessionManager.spawn(...)` → `whenDone`。原链路**信号完全没传给子会话**；`whenDone` 只订阅状态变化，不监听 abort。
- **死锁场景**：子 Agent 内部若卡在 `compact`/`complete`/`streamChat`（见 1/2/3），`runAgent` 永不 yield `done` → `whenDone` 永不 resolve → **主循环 delegate 调用永久挂起，整个程序卡死**。
- **修复**：`delegate.execute` 透传 `ctx.signal`；`SpawnOptions`/`runBackground`/`whenDone` 全部带 signal；`whenDone` 在 signal abort 时立即 resolve（返回当前会话），使主循环可被 Ctrl+C 打断。

### 🟠 5. MCP `connect` / `listTools` 无超时（启动期卡死）
- **位置**：`src/mcp/client.ts:30` `connect()`、`:46` `listTools()`；`cli/main.ts:118` `await toolManager.getAllTools()`
- **机制**：`connect()` 内部 `await this.client.connect(transport)` 无任何超时。
- **死锁场景**：配置了 playwright/github（docker 方式），若 Docker 未启动或 server 启动挂起，`getAllTools()` 在**启动阶段永久阻塞**，连 TUI 都进不去。
- **修复**：新增文件级 `withTimeout()` 辅助，给 `connect` 与 `listTools` 各加 30s 硬超时；超时后由上层 `provider.listTools` 的 catch 跳过该 server。

### 🟡 6. `run_command` 只杀 shell 不杀进程树（dev server 残留）
- **位置**：`src/tools/index.ts:322` `spawn(shell:true)` + `:335` `child.kill`
- **机制**：`child.kill` 只杀 shell 进程；`npm run dev` 等通过 shell 启动的 **vite 孙进程变孤儿**，继续占端口、持续运行。
- **后果**：非直接卡死循环，但下次 `run_command` 起同端口会 `EADDRINUSE` 失败，且后台残留进程占用资源（你"调试网页"场景里 dev server 可能一直后台残留）。
- **修复**：POSIX 用 `detached` + `process.kill(-pid)` 杀整个进程组；Windows 用 `taskkill /pid <pid> /t /f` 递归清理。**已用真实进程实测通过**（中断后 `ping -t` 孙进程被递归杀掉）。

## 二、已确认「安全」的循环相关点（无需修）

- **主循环无进展 / 重复 early-exit**：`loop.ts` 的 `noProgressStreak`（连 3 轮失败）+ `repeatCount`（连 3 轮相同工具调用）已防「模型陷入死循环烧 token」。
- **`maxIter=12` 上限**：物理限制总轮数，无无限 `while`。
- **工具结果配对 400 防护**：`history.ts:123` `getMessages()` 会修补「缺 tool 结果的 tool_calls」，避免 API 400 崩循环（上轮已修）。
- **`run_command` 自身**：已有 120s `timeout` + 输出 `MAX_OUTPUT=8000` 截断 + 流式消费 stdout/stderr（不会因管道满死锁）。
- **MCP `callTool`**：上轮已加 90s 超时 + signal + 连接自愈（resetConnection），本轮未改动。

## 四、给你的使用建议

- **现在整条链路都能被 Ctrl+C 打断**：主循环流式生成、上下文压缩、复合工具、子 Agent、MCP 调用，信号已全程透传。
- **仍建议**：调试网页时尽量别手动关浏览器窗口（playwright 会自愈重连）；长命令（如 `npm run dev`）用后台或确认能正常退出，避免端口残留。
- **启动前确保 Docker 在运行**（playwright/github 是 docker 起的），否则 30s 后会跳过该 MCP server 而非永久卡死。
